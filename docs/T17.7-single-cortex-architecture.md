# T17.7 — Single-Cortex Integration Architecture Plan

> Per Gee 2026-04-18: *"we are doing it all fully and no short cuts or jerry rigging shit this is unitys Brain so we have to be very careful in how it all works so the equations can work"*.
>
> This is the architectural migration plan for collapsing the separate `cortexCluster` (language cortex) into the main 201M-GPU `cortex` cluster as slice-indexed sub-regions. No code gets touched until this plan is clear and the equations are verified to be equivalent under the new layout.

---

## Current state (pre-T17.7)

Two parallel cluster stacks:

**Main-brain clusters** (on GPU via `compute.html`):
- `cortex`, `hippocampus`, `amygdala`, `basalGanglia`, `cerebellum`, `hypothalamus`, `mystery`
- Each stored in `server.clusters[name]` as `{size, spikeCount, firingRate, meanVoltage}` (server-side aggregate only — per-neuron state lives on GPU)
- Per-neuron state on GPU via `gpu-compute.js` buffers (`voltages` / `spikes` / `currents`)
- Drive uniform via `compute_batch` → `stepNeurons` runs LIF per substep
- Intra-cluster synapses empty (never uploaded — main brain runs without intra-cluster recurrence today; T18.4.a wired the plumbing, but no synapse matrix gets uploaded for main-brain clusters)
- No sub-regions — each cluster is homogeneous

**Language cortex cluster** (separate, `this.cortexCluster`):
- Full `NeuronCluster` instance in Node memory
- Has `regions` map: `{auditory, visual, free, letter, phon, sem, fineType, motor}` with `{start, end}` index ranges
- Has `synapses` (intra-cluster sparse CSR) + `crossProjections` (14 sparse CSR matrices)
- `lastSpikes` `Uint8Array` lives on server; updated by `cluster.step()` / teach methods
- GPU propagate via sparse-upload chunked binary frames — `_sparseMatrices["cortex_intraSynapses"]` + `_sparseMatrices["cortex_sem_to_motor"]` etc. stored by name
- Cross-projection Hebbian: CPU-authoritative + GPU shadow
- `cluster.step()` runs on server Node — CPU LIF integration with GPU-cached currents (T17.3.e one-tick-lag, T18.4.b full-await cascade)

**The two stacks do NOT share substrate.** Main-brain `cortex` and language `cortexCluster` are distinct objects with distinct neurons. The main-brain `cortex` is a large undifferentiated population; language happens in a separate, smaller (~7M) cluster with detailed sub-region structure.

## Target state (post-T17.7)

Single unified cortex on GPU:

- One `cortex` cluster, size ~201M neurons on 16 GB GPU (scales with hardware per `BRAIN_VRAM_ALLOC`)
- Language sub-regions are **slices** of the cortex neuron array at stable offsets:
  - `auditory: [0, 0.083·N)`, `visual: [0.083·N, 0.25·N)`, `free: [0.25·N, 0.5·N)`, `letter: [0.5·N, 0.55·N)`, `phon: [0.55·N, 0.75·N)`, `sem: [0.75·N, 0.917·N)`, `fineType: [0.917·N, 0.967·N)`, `motor: [0.967·N, N)`
  - Same fractional layout as today's `cortexCluster.regions`, scaled up to the full main-cortex `N`
- Intra-cluster synapse matrix uploaded to main-cortex GPU buffer (`bufs.synValues` / `synColIdx` / `synRowPtr`) via chunked binary upload
- 14 cross-projections become sparse CSR matrices indexed against main-cortex spike slices:
  - `sem_to_motor`: rows = `sem_size`, cols = `motor_size`, addresses slices at sem offset and motor offset
  - `cross-projection.propagateSparse` reads source-slice spikes from main-cortex `spikes` buffer at the source-slice offset, writes to a `postCurrents` buffer, then those currents get added to main-cortex `currents` buffer at the target-slice offset before next LIF step
- All curriculum teach methods retarget to: write pattern to GPU spike slice, dispatch GPU Hebbian on intra or cross matrix
- All generation/emission reads GPU spikes via atomic readback at slice offsets
- `cluster.step()` on server becomes thin — essentially dispatches the compute_batch + reads back aggregated results
- No separate `cortexCluster` object; language is an addressed SUB-STRUCTURE of the main cortex

### Why this works equationally

The Rulkov map, sparse matmul, LIF integration are all **pointwise per-neuron**. A slice of a 201M cluster running the same equations with the same local connectivity structure is mathematically equivalent to a standalone 7M cluster. The cross-projections connect sub-regions the same way whether the neurons sit in a standalone cluster or in a slice of a bigger cluster — they're just sparse matrices with the same (row, col, value) tuples. The equations do not change; only the indexing does.

The biological weights from `BRAIN_VRAM_ALLOC` don't change either — we're not adding neurons, we're moving the 7M language cortex into the 201M main cortex's existing budget. The language sub-regions consume ~7M of the cortex's 201M; the remaining ~194M neurons remain homogeneous cortex population doing predictive coding on sensory input.

### What changes, functionally

- Language cortex now runs **entirely on GPU** — no server CPU `cluster.step()` for language
- Curriculum teach dispatches sparse Hebbian and spike injection to GPU instead of writing to a `Uint8Array` in Node memory
- Readback for emission comes from GPU atomic reductions (`readbackVoltageMean` pattern, extended to slice-range)
- Eliminates the ~194M neurons of "dead" cortex that currently do nothing — language occupies the space productively
- One source of truth for cortex state (GPU); no split-brain between server-CPU language state and GPU main-brain state

## Migration phases (atomic, each one keeps the brain functional)

### Phase A — Foundation (no behavior change)

**A1.** Extend `gpu-compute.js uploadCluster` signature to accept optional `regions` metadata:
```
uploadCluster(name, size, voltages, synapses, lifParams, regions)
```
Where `regions` is `{auditory: {start, end}, visual: {start, end}, ...}`. When present, stored on `bufs.regions` for later slice-range operations. When absent, cluster is homogeneous (current behavior).

**A2.** Add GPU helpers for slice-range operations:
- `writeSpikeSlice(name, startIdx, endIdx, spikesArrayBuffer)` — stage spike pattern into GPU spike buffer at the slice range
- `readbackSpikeSlice(name, startIdx, endIdx)` — atomic-reduce spike count in slice range, return
- `writeCurrentSlice(name, startIdx, endIdx, currentsArrayBuffer)` — stage per-neuron current injection for a slice (used for sensory input + intent injection)

**A3.** Extend the sparse CSR upload path so a cross-projection sparse matrix can specify source-slice + target-slice offsets instead of standalone row/col sizes:
```
uploadSparseMatrix(name, rows, cols, values, colIdx, rowPtr, {srcCluster, srcSlice, dstCluster, dstSlice})
```
When `srcCluster` + `dstCluster` are specified, `propagateSparse` reads pre-spikes from `bufs[srcCluster].spikes` at `srcSlice` offset (instead of the standalone `preSpikes` buffer) and adds post-currents to `bufs[dstCluster].currents` at `dstSlice` offset (instead of its standalone `postCurrents` buffer).

**A4.** Add compute.html message handler `upload_cluster_with_regions` that calls the extended uploadCluster. Server can now ship the cortex cluster with language sub-region metadata.

**Phase A ships:** extended GPU substrate. No server code calls the new paths yet. Behavior unchanged; existing `cortexCluster` still works alongside.

### Phase B — Dual-cortex bridge (both cortices live, language still CPU)

**B1.** Server boot uploads main-cortex cluster WITH language sub-region metadata (Phase A extension). Main-cortex `cortex` now has the language regions registered on GPU — but no data written to those slices yet.

**B2.** At each compute_batch tick, after main-brain step completes, mirror the standalone `cortexCluster.lastSpikes` into the main-cortex GPU spike slice via `writeSpikeSlice`. Readback language sub-region spike count from main-cortex via slice-range atomic reduction.

**B3.** State broadcast now has TWO sources of truth for language sub-region activity (`cortexCluster.lastSpikes` on server + main-cortex GPU slice). They should match. Log divergence as a diagnostic. This is the equivalence verification — if the equations are correct, both should report identical spike counts per tick.

**Phase B ships:** main-cortex cluster carries language sub-region state redundantly. Divergence check validates the migration is equation-equivalent. `cortexCluster` is still authoritative for curriculum + generation.

### Phase C — Migrate curriculum teach methods one at a time

**C1.** `_teachWordEmission` — first target because it's the most complex (sem write + motor write + letter continuation chain). Rewrite to:
- Write sem slice spike pattern via `writeSpikeSlice('cortex', sem.start, sem.end, embBuffer)`
- Write motor slice spike pattern for expected letter
- Dispatch GPU Hebbian on main-cortex intra-synapse matrix (new upload via Phase A's chunked path) with pre/post vectors computed from slice offsets
- Dispatch GPU Hebbian on `sem_to_motor` cross-projection (Phase A.3 cluster-bound sparse)
- Mirror write to `cortexCluster.lastSpikes` for Phase B equivalence check during migration

**C2.** `_teachPhonemeBlending` — same pattern, retargeted to phoneme + letter slices.

**C3.** All other teach methods migrated similarly. Each commit migrates ONE teach method + the equivalence check log proves it still produces identical cortex state.

**Phase C ships:** curriculum runs on the unified GPU cortex. Standalone `cortexCluster` still exists but only gets its state mirrored (not driven) from teach methods. `cluster.step()` on server not yet removed.

### Phase D — Migrate generation / emission

**D1.** `cluster.generateSentence` — rewrite to run against GPU spike slices. Per tick: dispatch main-cortex step (already runs), readback motor slice via atomic-reduce for argmax decode. No server CPU cluster.step needed for language.

**D2.** `cluster.generateSentenceAwait` — same retarget.

**D3.** `languageCortex.generateAsync` — consumes the new GPU-sliced cluster interface.

**Phase D ships:** generation end-to-end on GPU. The separate `cortexCluster` object now has identical state to the main-cortex language slices (Phase B verifies) — time to delete.

### Phase E — Delete the standalone cortexCluster

**E1.** Remove `this.cortexCluster` from brain-server.js. Remove `new NeuronCluster('cortex', langCortexSize, ...)` construction. Remove `_cachedIntraCurrents` / `_cachedCrossCurrents` / `_dispatchGpuPropagates` from cluster.js since those were the CPU-side hack for a CPU cortex.

**E2.** Remove the mirroring code from Phase B. `cortexCluster.lastSpikes` deleted; everything reads from GPU.

**E3.** Persistence migration: `state.t14Language` block schema changes to reference main-cortex slice state. Version bump in persistence.js (VERSION 4 → 5) so stale saves reject.

**E4.** Legacy `NeuronCluster` class sticks around only for API surface (`readInput`, etc.) — wraps GPU slice accessors.

**Phase E ships:** single-cortex architecture live. No server-CPU cortex simulation. All language work runs on GPU.

### Phase F — Verification + docs

**F1.** Full Part 2 K-curriculum run at biological scale on the unified cortex — Gee-verified localhost.
**F2.** Docs updated: `docs/ARCHITECTURE.md`, `docs/EQUATIONS.md`, `brain-equations.html`, `unity-guide.html`, `README.md`, `SETUP.md`. Every reference to "language cortex cluster" or `cortexCluster` becomes "language sub-regions of the main cortex" or similar.
**F3.** Gee's signoff that equations still work correctly — required before push to main.

## Risks + mitigations

**Risk:** Equation drift — slice indexing off-by-one or slice-boundary bugs produce different spike counts vs standalone cluster.
**Mitigation:** Phase B mirrors state and logs divergence. Any non-zero divergence blocks Phase C migration.

**Risk:** GPU memory pressure — intra-cluster synapse matrix for main cortex at 201M neurons × fanout 1000 = 201B entries × 12 bytes = 2.4 TB. Won't fit.
**Mitigation:** Main-cortex intra-synapse matrix is SPARSE and connectivity-capped at `targetFanout / size`. At 201M neurons with targetFanout=300 (not 1000 — cortex is less densely recurrent than the full biological ratio), that's 60B entries × 12 B = 720 GB. Still won't fit. So language sub-regions run as slice-scoped intra-synapses: only the `letter × letter`, `phon × phon`, `sem × sem`, etc. sub-blocks get intra-synapses. The remaining 194M homogeneous cortex population has NO intra-synapse matrix at all (it doesn't today either — main brain currently has 0 intra-cluster synapses per the T18.4.a diagnosis). Sub-block synapses total: sum of (sub_size × targetFanout × 12 B) across 8 sub-regions = ~3 GB. Fits.

**Risk:** Curriculum teach throughput regression — moving CPU-direct `lastSpikes` writes to GPU `writeSpikeSlice` dispatches adds WebSocket round-trip per write.
**Mitigation:** Batch writes. A single teach-rep can batch all spike-slice writes into one `compute_batch_teach` message instead of N individual writes. Current pattern (one write per sub-region per word per rep) would be slow; batched pattern is one message per word.

**Risk:** Generation latency — `generateSentence` currently runs 2000 ticks on server CPU, reading `cortexCluster.lastSpikes` directly. Moving to GPU adds 2000 readbacks.
**Mitigation:** `readbackSpikeSlice` for motor region only (small — ~6.6M neurons × slice fraction 0.033 = ~220K spikes per readback, atomic-reduced). No voltage readbacks needed per-tick. Batched dispatch: server tells GPU "run N ticks, readback motor spike sequence", GPU runs local loop and streams back. Cuts 2000 round-trips to 1.

**Risk:** Curriculum save/load compatibility — existing saves have `state.t14Language.letterInventory` etc. bound to `cortexCluster`.
**Mitigation:** Persistence VERSION bump (5) auto-rejects pre-T17.7 saves. No migration code for old saves — curriculum re-runs from scratch on first boot after the bump. Gee's auto-clear-stale-state LAW already enforces this on every server restart anyway.

## Decision points that need Gee input before implementation

1. **Main-cortex intra-synapse matrix size.** Should the ~194M homogeneous cortex population outside the language sub-regions get an intra-synapse matrix? Would add ~2 GB VRAM for a sparse-300-fanout recurrent on that population. Currently the answer is NO (matches today's empty main-brain intra-synapses), but it's a design decision.

2. **Main-cortex sensory input wiring.** Today `injectText` on brain-server.js writes to a 1000-neuron slice of the standalone cortexCluster + amygdala. Post-T17.7, where does text injection land in the 201M unified cortex? Default: keep same 1000-neuron Wernicke's-area slice, but scaled to 201M × 0.001 = 201K neurons. Bigger sensory footprint matches biology but might dilute signal — design choice.

3. **Render boundary.** Does the 3D brain viz render language sub-regions as their own point-cloud populations (today's behavior from `brain-3d.js`) or as highlighted slices within the main cortex point cloud? Visual decision; either works.

4. **Should Phase B equivalence-check block Phase C automatically?** If divergence is non-zero for any tick, block the C migration. Strict = safer but slower to ship; loose = faster but more risk. Default: strict.
